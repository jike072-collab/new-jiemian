# Codex 工作规范

本规范吸收独立审查、最小改动和规格先行的工程方法，但只作为本项目内部规则使用。不复制外部项目代码，不引入新的插件依赖，不改变业务功能。

## 1. 3106 / 3107 规则

- 3106 是服务器正式环境，服务器最终只运行 3106。
- 3107 只在当前开发电脑运行，用于代码优化、自动测试和人工验收。
- 3107 不部署到服务器，也不需要服务器 systemd、Nginx 或数据目录。
- 所有服务器准备改动必须在 `chore/server-production-prep` 完成。
- 每个模块只能有一个主题明确的 commit，模块测试通过后立即 push 到 GitHub。
- `main` 代表允许进入正式发布流程的代码，但 Codex 不得自动合并 `main`。
- 最终流程是：模块分支开发 -> 本地 3107 测试 -> push GitHub -> 代码审查 -> 合并 `main` -> 服务器部署 3106。
- 服务器部署不属于本次自动任务。
- 禁止直接改动、重启、部署或覆盖 3106。
- 禁止直接 push 到 `main`。
- 发现问题时优先使用 `git revert` 回退独立 commit，不使用 `git reset --hard`、强制推送或改写公共历史。

## 2. Ponytail 最小改动原则

每次写代码前，Codex 必须先检查：

1. 这个功能是否真的现在需要？
2. 现有代码是否已经有类似能力？
3. 是否可以复用现有 helper、组件、API、数据结构？
4. 是否可以用浏览器原生能力、Node 标准能力或项目已安装依赖解决？
5. 是否可以只改一个小地方，而不是大范围重构？
6. 是否可以不新增依赖？
7. 如果必须新增代码，是否是最小可用改动？

禁止：

- 为小功能做大架构。
- 为测试阶段提前引入复杂系统。
- 为一个按钮新增一套框架。
- 重复实现已有逻辑。
- 新增无必要依赖。
- 牺牲安全、输入校验、错误处理、数据保护来减少代码。

## 3. Superpowers 阶段流程

每个阶段必须按这个顺序：

1. Understand：先说明理解到的需求。
2. Inspect：先检查相关文件和现有实现。
3. Spec：写出本阶段要达成的结果。
4. Plan：列出会改哪些文件、怎么改、如何验证。
5. Implement：只做本阶段范围内的改动。
6. Verify：运行 `npm run lint`、`npm run typecheck`、`npm run build`。
7. Review：输出结构化审查报告。
8. Push：推送到 GitHub 分支。
9. Wait：等待人工确认，不要直接合并 `main`。

## 5. 审查严重程度

### critical

- 泄露 API Key。
- 会破坏 3106 正式数据。
- 会删除 `uploads` / `data`。
- 会导致项目无法启动。
- 会绕过后台权限。

### high

- 改变现有 API 路径。
- 改变现有数据结构但没有兼容。
- 影响图片/视频生成主流程。
- 新增未说明依赖。
- 错误处理可能导致数据丢失。

### medium

- UI 行为变化但未说明。
- 文档和实际实现不一致。
- 缺少手动测试说明。
- 3107 / 3106 环境隔离说明不完整。

### low

- 文案问题。
- 注释不清楚。
- 文件组织可以更好。

## 6. 最小验证要求

每次提交前必须运行：

```bash
npm run lint
npm run typecheck
npm run build
```

如果失败：

- 不要提交。
- 说明失败命令。
- 贴出关键错误。
- 修复后重新运行。
- 全部通过后再 commit。

## 7. 禁止事项

- 禁止直接 push `main`。
- 禁止直接修改、重启、部署或覆盖 3106。
- 禁止把 3107 部署到服务器。
- 禁止自动合并 `main`。
- 禁止使用 `git reset --hard`、强制推送或改写公共历史。
- 禁止提交 `.env.local`。
- 禁止提交 API Key。
- 禁止提交 `data`。
- 禁止提交 `uploads`。
- 禁止提交生成图片、视频、用户上传文件。
- 禁止一次性大重构。
- 禁止没有计划就写代码。
- 禁止没有验证就说完成。
- 禁止为了“看起来高级”提前引入数据库、登录、支付、队列。

## 8. 允许事项

- 可以先写文档和流程。
- 可以做小范围安全修复。
- 可以增加错误提示。
- 可以增加日志，但不能记录敏感信息。
- 可以增加 3107 测试清单。
- 可以增加数据隔离说明。
- 可以整理代码，但必须小步进行。
- 可以为未来 SQLite、PostgreSQL、对象存储做规划，但不要提前实现复杂迁移。
